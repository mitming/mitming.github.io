Devils in BatchNorm
Created2020-10-24|Updated2020-10-25|BN
|Word count:2.9k|Reading time:10min

先看下普通的 BN, 在 batch 上对所有的 channel 做 normalization, 把下图中的蓝色部分 norm 为 0 均值 1 方差, 在 normalization 后还加了一个 channel-wise 的 affine 层, 可以看成是一个非常退化的卷积层, 加了后精度会稍微好点, 因为网络更深了, 但本质上没有影响.

对于 BN 来说, 最重要的还是 train 和 test 时的不一致性. 这种不一致性是由 BN 本身的定义导致的:

μB,σB2= mean, var⁡(x, axis =[N,H,W])\mu_{B}, \sigma_{B}^{2}=\text { mean, } \operatorname{var}(x, \text { axis }=[N, H, W]) μB​,σB2​= mean, var(x, axis =[N,H,W])

  • Training time:

x^=x−μBσB\hat{x}=\frac{x-\mu_{B}}{\sigma_{B}} x^=σB​x−μB​​

μEMA←λμEMA+(1−λ)μB\mu_{EMA} \leftarrow \lambda \mu_{E M A}+(1-\lambda) \mu_{B} μEMA​←λμEMA​+(1−λ)μB​

σEMA2←⋯\sigma_{E M A}^{2} \leftarrow \cdots σEMA2​←⋯

  • Test time:

x^=x−μEMAσEMA\hat{x}=\frac{x-\mu_{E M A}}{\sigma_{E M A}} x^=σEMA​x−μEMA​​

可以看到在训练时用这个 batch 的均值和标准差来norm, 但是测试时是没有 batch 概念的, 所以在原版的 batch norm 里面使用 EMA (指数移动平均)来做, 在训练时通过计算每一个 batch 的均值和方差, 再通过指数移动平均来计算 μEMA\mu_{EMA}μEMA​ 和 σEMA2\sigma_{EMA}^2σEMA2​, 在测试时用训练得到的 μEMA\mu_{EMA}μEMA​ 和 σEMA2\sigma_{EMA}^2σEMA2​ 做 norm.

BatchNorm 的好处有:

  1. 更好更快地收敛
  2. 对于初始化不好的模型也能让它 work
  3. 更加稳定的训练

但这种好处更多是由 norm 带来的, 而不是 batch norm 带来的, 后续的 Group Norm, Layer Norm 和 Instance Norm 也都取得了好的结果.

那么 Batch 给训练带来了什么好处? 就是 train 和 test 的不一致性, 这种不一致性既是好处又是坏处.

首先看下不一致性为什么会 work, 按照机器学习的思想, train 的时候是一个函数, 那么为什么 test 的时候可以用另一个函数呢? 这点可以用 dropout 或者 data augmentation 来解释.

Dropout 在训练时随机扔掉一半的神经元(p=0.5), 在测试时把所有数都除以 2, data augmentation 也是只在训练的时候做(也有 TTA).

可以认为 Dropout 在测试时做的事情是训练时的一个移动平均, 或者认为训练其实是在测试的基础上加了噪声.

在 BN 中使用的移动指数平均就是每个 batch 的平均, 所以 batch 的均值方差会和 moving average 的不一样, 这种不一样可以看出是加了噪声, 所以能够得出这种 inconsistency between training and testing 之所以能够 work 的原因:

  1. μEMA\mu_{EMA}μEMA​ 和 σEMA\sigma_{EMA}σEMA​ 能够近似 E[μB]E[\mu_{B}]E[μB​] 和 E[σB]E[\sigma_{B}]E[σB​]
  2. μEMA\mu_{EMA}μEMA​ 和 σEMA\sigma_{EMA}σEMA​ 是 EMA 添加了微小噪声的结果, 因为每个 batch 的均值和方差和 μEMA\mu_{EMA}μEMA​, σEMA\sigma_{EMA}σEMA​ 是不一样的, 所以可以看成是为移动平均 μEMA\mu_{EMA}μEMA​, σEMA\sigma_{EMA}σEMA​ 添加了一个噪声 μB\mu_{B}μB​ 和 σB\sigma_{B}σB​
  3. 看成和 dropout 或者数据增强一样, 测试是训练的一个移动平均

那么肯定就能得出这种不一致性在哪些情况下不会 work, 当测试时的均值和方差 μEMA\mu_{EMA}μEMA​ 和 σEMA\sigma_{EMA}σEMA​ 不能近似训练集真实的均值和方差 E[μB]E[\mu_{B}]E[μB​] 和 E[σB]E[\sigma_{B}]E[σB​] 的时候, 即此时 μEMA\mu_{EMA}μEMA​ 和 σEMA\sigma_{EMA}σEMA​ 不能看成是 μB,σB\mu_{B}, \sigma_{B}μB​,σB​ 的一个指数移动平均.

这种不 work 的情况在使用时具体表现在:

  1. EMA 计算错误
  2. 每个 batch 的均值和方差不稳定, 换句话说是模型 和/或 数据的不稳定

来看下 EMA 具体的一些问题:

μEMA←λμEMA+(1−λ)μB\mu_{EMA} \leftarrow \lambda \mu_{E M A}+(1-\lambda) \mu_{B} μEMA​←λμEMA​+(1−λ)μB​

σEMA2←⋯\sigma_{E M A}^{2} \leftarrow \cdots σEMA2​←⋯

根据公式不难看出什么时候 EMA 不是 E[μB]E[\mu_{B}]E[μB​] 和 E[σB]E[\sigma_{B}]E[σB​] 的一个近似:

  • λ\lambdaλ 很小的时候, 这个情况会导致出现 EMA biased, EMA 很大程度会由最近的那个 iter 的均值和方差所决定, "忘掉"了之前 batch 的均值和方差, 所以一般取值在 [0.9, 0.99]
  • λ\lambdaλ 很大的时候, 每个 batch 对 EMA 的贡献很小, 所以需要有很多的 batch/iteration 才能让 EMA 收敛到每一个 batch 的均值和方差的平均.
  • 模型/数据不稳定, 具体说是模型/数据产生的每一个 batch 的均值和方差不稳定

一旦 EMA 不 work, 那么很容易产生 “false overfitting” 的现象, 看上去在训练集上效果很好, 测试集上效果不好, 好像是过拟合了, 但实际上是因为 EMA 只在 test 的时候使用. 可能是训练的 iteration 太少了, 所以没有个好的 EMA 给模型在 test time 的时候使用.

可以看到在 BN 中使用 EMA 有以下几个问题:

  1. Always biased, 移动指数平均决定了永远是最近的 iteration 对 EMA 有最大的权重, 会渐渐忘掉之前的 iteration.
  2. Always estimated on non-stationary data, 在每一个 iteration 时候模型的参数都会变, 所以每个 batch 的均值和方差也会变, 所以 EMA 永远是作用于变化的数据上, 所以自然不靠谱.
  3. just a cheap vision of “true average”

所以不如直接计算真正的 Average, 而不是使用指数移动平均, 这种方式叫做 Precise BatchNorm

计算方法是用 n 个 iteration 计算真正的 E[μB]E[\mu_{B}]E[μB​] 和 E[σB]E[\sigma_{B}]E[σB​], 而不是 moving average, 并在 test time 的时候使用. 这个东西其实最早在 ResNet 中就有说了, 但一直没火起来, 原因在于很多情况下, EMA 是 work 的, 因为 λ\lambdaλ 一般都会设置的比较大, 模型训练时间也比较长, 训练到最后 lr 会越来越小, 导致模型也训练会越来越稳定, 这样在模型的训练最后 EMA 会比较好地近似 E[μB]E[\mu_{B}]E[μB​] 和 E[σB]E[\sigma_{B}]E[σB​]

上图可以看到在整个训练过程中, batch size 的大小对训练过程影响不是很大, 不但训练结果没啥区别, 而且训练过程的 training curve (2 个实线)也非常相似, 但是 validation curve 影响看起来还是比较大的, 在前面的时候差距比较大, 直到后面才比较接近, 而且用 8k 的 batch size 训练的时候 validation curve 非常地不稳定, 看起来像是过拟合, 其实就是 “false overfitting”. 这就是 EMA 不稳定的情况.

为什么会出现这种情况? 因为 batch size 大了 32 倍, 相应地在一个 epoch 中训练的步数就少了 32 倍, 上面说过当步数少的时候, EMA 不能近似真正的平均, 同时根据线性准则, 学习率也大了 32 倍, 那么模型变化程度也变大了 32 倍, 前面也说了当模型变化程度太大了的时候, EMA 也无法近似真正的平均. 直观上理解, 模型一直在变, EMA 反应的是历史上的这些 iteration 里的 mean 和 std, 由于模型变得太快(学习率增大), 最后用于测试的模型已经和历史学习的模型不太一样了, 也会导致出问题.

作者在 Group Norm 的期刊版本里面做了个实验:

可以看到用了 Precise BN 后 8k batch size 的 curve 和小 batch size 的 curve 几乎都是重叠的

总结下什么时候用 Precise BN: 在 inference 且 EMA 不稳定的时候, EMA 不稳定是指模型没有在稳定的状态下保持充分的迭代次数.

一个简单版本的 precise bn 是继续使用 EMA 但是把模型固定住, 就是训练一定 iteration 后模型不再 train 了, 然后直接跑 1000 个 iteration 的 forward, 由于只固定了模型, 所以 EMA 还是会继续更新, 这种做法肯定比原来的 EMA 要好, 很多时候这种做法结果就已经很好了. 当然固定模型 train 1000 次得到真正的 average 理论上肯定会更好.

上面讲的是在 test/inference 时的坑, 下面说的是在 training 中的坑, 问题是 normalization batch size != SGD batch size, 就是来拿来 norm 的 batch size 和拿来做梯度下降的 batch size 可能是不一样的, 主要是历史原因, 大多数的 bn 实现都是单卡自己做 bn, 假设 SGD batch size = 64, 一共 8 卡, 那么每个卡都会分到属于自己的 batch size = 8, 每个卡只会对分到自己卡上的那个 batch 去算均值方差做 norm, 目前有 Sync BN, Ghost BN 和 Virtual BN 来解决这个问题.

在 ImageNet in 1 hour 这篇文章中说了 batch size 变大 32 倍的同时, lr 也要变大 32 倍, 但在文章中也强调了在这个过程中需要保持 “normalization batch size” 不变, 这里调了下 normalization batch size, 看看会发生什么.

在训练中一个 batch 的均值和方差可以看做是 test time 中的均值方差添加了 noise, 因为直观的想法是用一个 batch 均值和方差来代替总体(所有数据)的均值和方差, 所以当 batch size 小的时候会有更大的噪声, batch size 大的时候噪声会更小. 当然 noise 也是有用的, 可以类比于 dropout, 算是一种正则化, 减少模型过拟合, 增强泛化性.

从上图也可以看出, NBS(normalization batch size) 越大时 train error 越小, 但 val error 却不是这样, 而是取一个中间值的时候效果最好.

为了增大 normalization batch size, sync bn/cross-gpu bn 把很多 GPU 上的小 batch 合在一起变成一个大的 batch, 计算这个大 batch 的 mean 和 std, 以此来增大 normalization batch size. 这个适用于比较大的任务, 也就是每张卡分配的 batch size 比较小的情况.

为了减小 normalization batch size, 可以用 Ghost BatchNorm, 先把大的 batch split 到小的 batch, 计算完了再连回去, 这样也不需要改变单卡的 batch size.

还有一种方法叫 accumulate gradients, 这种方法能保持 normalization batch size 不变, 但是改变梯度下降的 batch size, 做法就是累积一定的 iteration 之后再去更新模型参数, 而不是每个 iteration 结束后都去更新.

当 noise 比较大的时候, 也可以用 batch renormalization 来解决, 是 Google 对 batch norm 的一个 follow up.

数据可能会出现 non-i.i.d(非独立同分布), 通常在以下几种情况下出现

  1. 当数据来源不一致的时候(在多个数据集上训练, fine-tuning), 这种情况可以用一些 trick:
  • Training: “Separate BN” for each domain
  • Training/fine-tuning: Frozen BN(不是 norm, 只是线性层, 是不会帮助网络收敛的, 所以一般在 fine-tuning 使用, 也可以在 training 时最后再用)
  • Testing: Adaptive BN
  1. GAN
    GAN 本身就有 2 种分布, real/fake
  2. 任务本身的原因
  • two-stage object detection, 可能一个 batch 中很多的 proposals 都来自同一张 image.
  • video understanding 中, 一个 batch 里面的很多 frames 可能来自同一个 video.

这些情况可以尝试使用上面说的 Ghost BN, Batch Renormalization 或者其他一些与 batch 无关的 norm, 例如 group norm 来解决.
